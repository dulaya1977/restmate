Workflow / Standard sequence

Recommended month-start sequence (10 steps)

The most important order is always: import previous month first, switch Report Month next, and only then start filling the new month.

Open RestMate.exeConfirm header and tabs load correctly.
Import previous month JSONLoad the latest confirmed file from last month to seed buffer data.
Switch Report Month in SetupChange to the target month to rebuild the month date range.
Press Refresh Data / Clear SettingReorganize crew status mapping and clear month-specific voyage settings.
Verify Setup fieldsVessel Name, IMO No., Prepared By, National Holidays, and Crew List.
Fill Voyage ControlEnter UTC Offset per ship date and handle timezone or IDL events.
Fill Crew GridEnter x/c/w/t marks by crew and date at 30-minute resolution.
Review personal and summary reportsCheck Personal Monthly, Summary, and OT Summary before printing.
Resolve Exceptions and ChecksFix missing values and warnings before final output.
Export final JSON backupSave and hand over the final state for traceability.
Critical: skipping Step 2 can break month-start rolling 7-day calculations due to missing pre-month history.
Same-month continuation: if you are only continuing the same month, import that month JSON and continue directly.
Kick Off restriction: RestMate blocks kick-off usage for previous month or earlier records.

Crew status and limits

  • On Board: most common full-month status.
  • Sign On: pre-sign-on cells become automatic sign markers.
  • Sign Off: post-sign-off cells become automatic sign markers.
  • Crew count range is 23 minimum to 46 maximum.

Refresh Data / Clear Setting behavior

  • Converts eligible Sign On rows to On Board.
  • Removes eligible Sign Off rows.
  • Fills blank status rows to On Board when crew data exists.
  • Remaps crew IDs and clears report-month voyage inputs.

Export first. This action clears UTC offset/event/affected time inputs for report-month dates.

Voyage Control / UTC and day length

Voyage data is the base for 24h and 7-day compliance math

Each row is one vessel-level ship date. UTC changes alter day length. IDL skip/repeat must be explicitly reviewed.

Field logic

  • In Report: YES for report month, BUFFER for 7 pre-month dates.
  • UTC Offset: supports integer and half-hour values.
  • Auto Event: inferred from offset deltas.
  • Event Override: use only for special manual cases.
  • Affected Time: start hour impacted by timezone event.

Practical offset filling

  • Enter the first valid offset, then only change on actual timezone shift dates.
  • +8 -> +9 usually becomes CLOCK_FORWARD with 23-hour day.
  • +9 -> +8 usually becomes CLOCK_BACKWARD with 25-hour day.
  • Blank offset produces missing state and check warnings.
  • IDL_SKIP day length is 0; IDL_REPEAT creates repeat-date behavior.

Crew Grid / daily entry

Use 30-minute marks for precise rest/work records

Crew Grid captures per-crew, per-date activity. Timezone and IDL affected cells should be reviewed first.

Mark definitions

  • x: Rest
  • c: Comp Rest
  • w: Work
  • t: OT
  • -: automatic sign marker (non-applicable slots)

Cycle order: x -> c -> w -> t -> x.

Quick operations

  • Copy Previous Day: duplicate same crew previous date.
  • Copy Selected Day: store currently selected grid snapshot.
  • Paste Copied Day: apply copied snapshot to current target.
  • Clear Day: reset selected day to default rest grid.
Reports and checks / pre-delivery verification

Treat checks as mandatory before print or handover

After filling, review all report layers and clear unresolved issues in Exceptions and Checks.

Personal Monthly

Single-crew month review with ≥6H, ≥10H, Σ7D, ≥77, plus OT/Comp context.

If #1, #2, or #3 shows ! or X, compensatory rest is required.

Summary

Whole-vessel daily aggregation for monthly filing and department review.

OT Summary

Aggregated OT values with timezone/IDL statistics and notice context.

Exceptions

Root-cause list for rest-hour exceptions, especially when you see ! or X.

Checks

Data integrity guardrail. Includes month setup, offsets, buffer rows, comp rest, and warning counts.

Expected quality gate

  • Critical X items resolved.
  • All ! items reviewed and explainable.
  • Printed outputs verified against final data state.

Compensatory rest mechanism

  • Trigger: any ! or X on #1, #2, #3 in Personal Monthly.
  • Daily deficit: check DFCT (Deficit); a negative value is the hours to recover for that day.
  • Monthly deficit: check Total DFCT (Total Deficit); a negative value is the monthly total to recover.
  • Recovery source must be future surplus rest after the violation date, not earlier surplus rest.
  • Surplus rest means daily rolling 24h rest above 11h; 11h or below gives zero surplus rest for recovery.
  • Recovery complete when OA BAL (Overall Balance) returns from negative to 0.
JSON / backup, import, merge

Use JSON operations to keep work safe and auditable

Consistent export habits prevent data loss. Import and merge should always be followed by checks.

Export JSON

  • Default name: ot33-YYYY-MM.json.
  • If month is not set yet: ot33-backup.json.
  • Export after each session and before major actions.
  • Keep versioned files for rollback and handover evidence.

Import JSON month-start routine

  • Import previous month final JSON first.
  • Switch Report Month to the new month.
  • Press Refresh Data / Clear Setting.
  • Refill voyage offsets and verify Pre-month buffer checks.

Import replaces the current on-screen state. Export before importing.

Merge JSON (two-source consolidation)

Use when two teams filled the same vessel/month separately. Merge is strict and rejects incompatible inputs.

  • Required: same IMO No., same Report Month, valid RestMate JSONs.
  • Rejected if merged active crew would exceed 46.
  • Rejected if first two crew records are identical (merge not needed).
  • Output file: ot33-YYYY-MM-merged.json and immediate apply to current view.
  • Voyage data comes from the primary source (first merged source), so re-check Voyage and Checks after merge.

FAQ / common blockers

Most frequent questions

Why are there BUFFER dates before month start?

RestMate uses the previous 7 days to calculate rolling 7-day compliance at the beginning of the month. BUFFER dates are for calculation and are not printed as official report dates.

Do I always need to import previous month first?

For formal month roll-over, yes. This provides the required source for pre-month offsets and crew-day grids. Without it, month-start rolling checks can be incomplete.

Why did one UTC offset change cascade to later dates?

This is expected behavior. RestMate assumes the same offset continues until the next timezone change date.

Why does Checks show Auto buffer import warning?

Usually the local previous-month snapshot is missing or incomplete. Import the prior month JSON manually and check buffer offset/grid status again.

When is compensatory rest required?

Use a simple trigger rule: if #1, #2, or #3 shows ! or X (linked to ≥6H, ≥10H, Σ7D), the record is non-compliant and compensatory rest is required.

How much compensatory rest is needed?

Use both fields: DFCT for daily deficit and Total DFCT for monthly deficit. Any negative value is a deficit that must be recovered.

What is the difference between DFCT and Total DFCT?

DFCT shows how many hours are missing for that specific day (if negative). Total DFCT shows the month-level deficit/surplus; if negative, the month is still short.

Can earlier surplus rest be used to offset a later violation?

No. Recovery must come from surplus rest generated after the violation occurs. Earlier surplus rest cannot be used to backfill a later non-compliant day.

What counts as surplus rest for compensation?

Surplus rest is only the portion above 11 hours in the day's rolling 24h rest total. If a day is 11 hours or below, it contributes no surplus rest for compensation.

When is the deficit considered fully recovered?

When OA BAL (Overall Balance) moves from negative back to 0, the deficit is considered fully compensated.

Why were Crew IDs changed after merge?

Merge reorders and remaps crew IDs so grids and notes from both files map correctly in one dataset. Use merge preview and final Crew List as source of truth.

My data disappeared after import. Is it lost?

Import replaces the current in-memory view with selected JSON content. Export before import to keep a recoverable copy.
